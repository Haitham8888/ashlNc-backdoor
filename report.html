Executive Summary

This report details a previously undocumented backdoor identified as ashlNc (named after the directory in %APPDATA%). The malware is a sophisticated, multi-layered Node.js backdoor that establishes persistence via Windows Registry and Scheduled Tasks, exfiltrates browser credentials and cryptocurrency wallets, and communicates with its operator through a novel three-stage bootstrap combining Solana blockchain memos, BitTorrent Kademlia DHT, and Socket.IO WebSocket.

The payload is triple-obfuscated (Base64 + AES-256-CBC + Webpack bundle, 747KB decompressed). It incorporates bittorrent-dht, adm-zip, socket.io-client, and NaCl cryptographic libraries (XSalsa20, Ed25519, Blake2b). The C2 infrastructure uses no fixed servers — the operator publishes DHT node IPs as Solana memo transactions, making takedown exceptionally difficult.

During forensic analysis, two additional infections were discovered on the same system: (1) a Nemotrix browser hijacker redirecting to www.nemotrix.org, and (2) a sophisticated Chrome extension (disguised as "Google Docs Offline") that performed keylogging, clipboard capture, Bybit cryptocurrency exchange cookie theft, and DOM snapshot exfiltration — all installed 3 weeks before the main backdoor.

The infection originated from a trojanized npm project downloaded on 2026-04-05. Running npm install triggered a supply chain attack that deployed the backdoor on 2026-04-08. A GitHub Security Advisory (GHSA-hxf3-5h5r-xr7c) has been published, and a CVE ID is pending from GitHub's CNA program. This is a zero-day backdoor — no prior public reporting exists for ashlNc or any related indicators.

Confirmed Infection Vector: npm Supply Chain Attack

Vector — npm Supply Chain (CONFIRMED) ROOT CAUSE

The victim extracted a trojanized npm project to their Desktop on 2026-04-05 at 08:37. This was a Solid.js/Vite web application project with an npm package.json containing dependencies including @supabase/supabase-js, cropperjs, imagekit, imagekit-javascript, node-fetch, pg, solid-js, and others. When the user ran npm install, one or more of these packages (or their sub-dependencies) contained malicious postinstall scripts that wrote the backdoor files to %APPDATA% and %LOCALAPPDATA%.

The 3-day gap (April 5 extraction to April 8 file creation) suggests either: (a) a delayed-execution mechanism in the malicious package, (b) the backdoor was part of the node_modules and was executed when the dev server was first started, or (c) the user ran a specific command that triggered the payload.

The bundled node.exe (v22.9.0) in the backdoor is the same Node.js version commonly installed via npm. The project's src/App.jsx (92KB) and src/Dashboard.jsx (193KB) are exceptionally large for JSX — may contain embedded obfuscated payload.

Related: npm supply chain attacks — [2], [3]

SECONDARY INFECTION — Chrome Extension (2026-03-11)

A separate and likely earlier infection installed a malicious Chrome extension at C:\Users\Tommy\AppData\Local\Google\Chrome\jucku\ on 2026-03-11. This extension was disguised as "Google Docs Offline" but had permissions for cookies, history, bookmarks, proxy, clipboardRead/Write, webRequest, and <all_urls>. This is a different operator from the ashlNc backdoor.

Command & Control — Decentralized Bootstrap

The C2 architecture is decentralized, using a three-stage bootstrap with no hardcoded IP addresses. This represents a significant evolution in malware resilience, making traditional IP-based blocking ineffective.

Stage 1 — Solana Blockchain (Novel technique)

getSignaturesForAddress("BjVeAjPrSKFiingBn4vZvghsGj9KCE8AJVtbc9S8o8SC")
Queries the Solana blockchain for memo transactions from wallet address BjVeAjPrSKFiingBn4vZvghsGj9KCE8AJVtbc9S8o8SC. Each memo contains a JSON payload with a link field — a base64-encoded DHT node IP address.

The blockchain is used as an immutable, censorship-resistant DNS. The attacker publishes new DHT nodes via Solana memos.

Stage 2 — Kademlia DHT (BitTorrent)

Bootstrap nodes: dht.libtorrent.org:25401, router.bittorrent.com:6881, router.utorrent.com:6881
Registers with the BitTorrent DHT network and looks up PUBLIC_KEY = ea1b4260a83348243387d6cdfda3cd287e323958 (160-bit infohash) to retrieve the live C2 configuration.

The C2 config is stored on the DHT under this infohash. No central server — the config can be updated by republishing to the DHT.

Stage 3 — C2 Configuration Structure

The DHT value returns a JSON object containing:

{ "_IP": "C2 HTTP server address", "_URL_FOR_LOG_":"AES-encrypted exfiltration endpoint", "_ASAR_ARHIVE_":"URL for additional payload (.asar archive)", "_ASAR_KEYS_": { "node_hash": "...", "node_key": "...", "node_iv": "..." }, "_PORT": "WebSocket port" }

The stolen data is first compressed using adm-zip.addLocalFolder(), then encrypted and exfiltrated. The AES key is transmitted in the HTTP header — a design choice that sacrifices key secrecy for simplicity.

Prevention — How to Avoid npm Supply Chain Attacks

The root cause of this infection is an npm supply chain attack — a malicious package (or compromised dependency) that executed a postinstall script. Here's how to protect yourself:

1. Audit Before Installing
npm audit / npm doctor

Always run npm audit before installing packages. Use npm install --dry-run to preview what will be installed. Check packages on socket.dev for risk scores.

2. Lock Your Dependencies
package-lock.json integrity

Commit package-lock.json to version control. Use npm ci instead of npm install in production — it respects the lockfile and fails if integrity doesn't match.

3. Disable Postinstall Scripts
npm config set ignore-scripts true

Postinstall scripts are how most npm malware deploys. Set ignore-scripts=true globally, or use npm install --ignore-scripts for untrusted projects. Only re-enable for trusted packages.

4. Sandbox Node.js Projects
Docker / VM for development

Run npm install inside a Docker container or VM for any project you don't fully trust. This contains the blast radius of any malicious package to the container.

5. Verify Package Signatures
npm registry signatures

npm packages can be signed. Use npm audit signatures to verify. Check the package's GitHub repo for suspicious activity (recent commits, new maintainers, etc.)

6. Monitor Your System
Sysmon + Windows Event Logs

Enable PowerShell logging (Script Block Logging) and Sysmon to detect suspicious child process creation (e.g., node.exe spawning PowerShell). Monitor %APPDATA% and %LOCALAPPDATA% for unexpected executable files.

Specific to this infection: The compromised project was a Solid.js/Vite application from an untrusted source. Always verify the authenticity of any project you clone or download — check the commit history, maintainers, and scan with npm audit before running npm install.
